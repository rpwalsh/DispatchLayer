# Data Policy

This policy defines how DispatchLayer data must be captured, stored, transformed, and presented.

## Policy Goals

- Keep data usage honest and traceable
- Prevent deceptive fallback behavior
- Preserve reproducibility for engineering and operations
- Protect proprietary and sensitive information

## Data Classes

1. Raw capture artifacts
- location: `data/raw`
- purpose: reproducible source evidence

2. Normalized snapshots
- location: `data/source_snapshots`
- purpose: local development and demo realism

3. Runtime API payloads
- generated by route logic from code + source inputs
- purpose: operational consumption by dashboard and clients

## Mandatory Requirements

- Never commit secrets or credentials
- Keep all internal timestamps in UTC
- Preserve source attribution through transformations
- Keep units explicit where practical
- Do not fabricate missing observations

## Quality and Missing Data Rules

If data is stale, missing, or partial:
- mark it explicitly in payload/state
- include warnings for consumers
- reduce confidence/trust appropriately
- avoid replacing missing critical fields with synthetic defaults

## Provenance and Traceability

Every meaningful transformed output should retain:
- origin provider/connector identity
- timestamp context
- quality/degradation context if present

Where available, attach decision trace metadata for post-event review.

## Reproducibility Standards

- Use scripts in `scripts/` for data capture refresh
- Keep source filenames stable when possible
- Record capture time and source context
- Prefer deterministic transformations for testability

## Access and Handling

This repository is proprietary preview software.
Data and derived outputs should be handled according to internal access controls and compliance policies.

## Review Cadence

Review this policy when:
- new provider/connector is added
- major schema changes occur
- incident analysis shows data-quality blind spots
